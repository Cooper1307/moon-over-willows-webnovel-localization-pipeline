## 四、输出格式模板

每章生成一个 `第X章术语与人设.md` 文件，保存到 `章节原文/第X章/` 目录。

严格遵循以下模板结构：

```markdown
# 第X章 术语库 & 角色人设档案

> 章节标题：[从原文第一行提取]

---

## 一、本章术语提取

### [分类名称]（如：人名与称谓 / 官职与机构 / 地点与居所 / 文化特色 / 婚姻法律 等）

### [角色名]

| 维度 | 详情 |
|------|------|
| **身份** | ... |

| **语言风格** | ... |

1. **[挑战名称]**：具体描述及建议处理方式

```

### 格式规范要点

1. **状态标记**：
   - `[已有]`：该术语在 `.agent/skills/translation_sop.md` 或前序章节中已有记录
   - `[新增]`：本章首次出现的新术语
2. **分类灵活**：根据每章内容实际出现的术语类型灵活分组，不必强求所有章节分类一致
3. **角色档案**：新角色需完整记录，老角色仅记录本章新增的性格/行为/状态变化
4. **翻译挑战**：标记诗词、文化梗、打戏、语调碰撞等需要人工重点处理的段落

## 五、核心约束法则：基于“动态切片索引”的先检索后输出机制 (CRITICAL)

为了保证横跨百章的本地化名称对齐与设定统一，且规避超长文件撑爆上下文的致命问题，接手本任务的 AI（您）必须严格遵守以下 RAG 提取策略：

1. **刷新切片数据库**：每次您接到一批章节提取任务时，由于之前可能有新增的内容，第一步您必须先在终端执行工作流指令 **`/build_knowledge`**。该指令会帮您生成与剔重所有历史档案。
2. **下载目录纲要**：脚本执行完毕后，使用 `view_file` 读取项目专属核心缓存文件 **`.agent/knowledge/Knowledge_Index.md`**，以获取总角色表册和术语类目表。*(注意，不要去读全量文件！)*
3. **查旧案定新词**：
    - 读取待处理的《第X章原文.md》后，在自己思路上挑出拟定的新条目。
    - 在动笔写出最终 Markdown 提取表格前，对于你圈出的可疑名单，必须运用 **`grep_search` 工具** 在全区 **`.agent/knowledge`** 里进行模糊追溯。（比如 `grep_search("千里寻", ".agent/knowledge")`）
    - **如果命中已有碎片**：务必继承查到的那份建议英文，并在现在的表里标注状态 `[已有]`；
    - **如果确属全书首现**：方可运用 AI 和翻译策略创造本地化词语，标注状态 `[新增]`。

## 六、防患未然的重点注意事项 (Anti-Hallucination & Consistency)

1. **绝对名称唯一原则（防异名分裂）**：在提取新的角色设定时，输出角色的名称**严禁**携带任何修饰语、括号或身份标注。比如，如果该角色叫“赵朴”，不管是他在当前章被称作“仪王”还是“少爷”，你在表格标题里**只能写 `### 赵朴`**！任何带有 `赵朴 (仪王)` 的自作主张写法都会导致脚本在后台生成重名的碎片卡片。
2. **严苛的 RAG 思考链 (CoT) 约束**：AI 禁止凭借自己跨文件的伪记忆强行断定一个词是 `[已有]` 还是 `[新增]`。**在你将任何一个词语标记为 `[已有]` 之前，你必须通过 `grep_search` 等搜索工具确切查到包含该词语碎片的记录文件**，否则必须打上 `[新增]` 重新翻译！
3. **角色的累积性**：新章若有旧角色登场，你在生成的表格里只需要写这个角色**在本章里出现的新增行为/特质**即可，千万不需要重复把索引库里写好的旧背景誊写进来。
4. **诗词与文风萃取**：任何出现的古诗词、词牌名、引用的文献，都要完整摘录放入“本章翻译挑战”区！因为每次您编译后，后台会将全书所有的难点抽取为专属参考书！
